# lead-profiler-haskell

A standalone Haskell library that hardens the JSON-parsing layer of the
[`lead-profiler`](../lead-profiler/web-app) Next.js app.

The original TypeScript API routes — `analyze-lead-images`,
`prospect-content-audit`, and `read-portfolio` — all rely on `JSON.parse`
plus a fragile `{ ... }` substring fallback to digest VLM/LLM output.
This library replaces that with:

- A **lenient pre-processor** that strips markdown fences, removes trailing
  commas, normalises smart quotes, quotes bare object keys, converts
  single-quoted strings to double-quoted, and strips zero-width characters.
- **Strict algebraic-data-type schemas** for all three routes, parsed via
  `aeson` with field-level error accumulation.
- A **defaults layer** that fills in missing or `unknown` fields with honest
  sentinel values, so downstream code never has to handle `undefined`.
- A **code generator** that emits TypeScript types, JSON Schema, and sample
  fixtures from the same ADTs — single source of truth, no schema drift.

The deliverable is a **library + test suite** proving the pipeline correctly
digests a corpus of broken LLM outputs. There is no service layer and no
Next.js integration — that's a follow-up.

## Build & run

```bash
# Requires GHC 9.10+ and cabal-install 3.10+
cabal build       # build the library + CLI
cabal test        # run the test suite (102 examples)
```

### CLI usage

```bash
# Emit TypeScript types (drop-in replacement for lib/lead-profiler-types.ts)
cabal run lead-profiler-haskell -- emit-ts > generated/lead-profiler-types.ts

# Emit JSON Schema (Draft 2020-12)
cabal run lead-profiler-haskell -- emit-schema > generated/schemas.json

# Emit sample fixtures
cabal run lead-profiler-haskell -- emit-fixtures > generated/fixtures.json

# Validate raw LLM output against a route (reads from stdin)
echo '{"primary":"x","secondary":"","past_work_examples":[],"track_record":"x","service_depth":"x","extraction_notes":"x"}' \
  | cabal run lead-profiler-haskell -- validate read-portfolio
```

   ▼

## What the lenient pre-processor fixes

The original TS code in each route handler was:

```typescript
let stripped = content_text.trim();
if (stripped.startsWith("```")) {
  stripped = stripped
    .split("\n")
    .filter((l) => !l.trim().startsWith("```"))
    .join("\n")
    .trim();
}
try { extracted = JSON.parse(stripped); }
catch {
  const start = stripped.indexOf("{");
  const end   = stripped.lastIndexOf("}");
  if (start >= 0 && end > start) {
    extracted = JSON.parse(stripped.slice(start, end + 1));
  } else {
    return NextResponse.json({ error: "Could not parse VLM response as JSON" }, ...);
  }
}
```

This is fragile in four ways — each is fixed by `LeadProfiler.Lenient`:

| Problem                                            | TS behaviour                                | Haskell fix                                    |
|----------------------------------------------------|---------------------------------------------|------------------------------------------------|
| Code fences prefixed by prose                      | Fence-strip pass only fires if input starts with @```@. Prose slips through and breaks @JSON.parse@. | `stripCodeFences` strips fence lines anywhere in the input, not just at the start. |
| `indexOf('{') ... lastIndexOf('}')` heuristic      | Captures the /outermost/ braces — wrong if the model emitted two adjacent objects, or follow-up prose containing `}`. | `findBalancedJsonObject` does a proper brace-depth scan that respects string literals. |
| Trailing commas, smart quotes, bare keys, single quotes | All defeat `JSON.parse`.                  | Six normalisation passes fix each issue, and each pass emits a warning so the caller knows what was repaired. |
| No diagnostics                                     | Errors are just `"Could not parse VLM response as JSON"` with no clue /why/. | `ExtractWarning` (8 constructors) + `ValidationError` (4 constructors) give the caller enough context to surface a useful error to the user. |

### Warnings

Each normalisation pass emits a warning when it actually changes the text:

- `HadLeadingOrTrailingProse` — stripped non-whitespace text around the JSON object
- `HadCodeFence` — stripped a markdown code fence
- `HadTrailingComma` — removed a trailing comma
- `HadSmartQuotes` — normalised typographic quotes
- `HadSingleQuotes` — converted single-quoted strings
- `HadBareKey` — quoted an unquoted object key
- `HadZeroWidthChars` — stripped zero-width / BOM characters
- `UsedBalancedSliceFallback` — the trivial first-`{`...last-`}` heuristic failed; fell back to a balanced-brace scan

## Test results

```
cabal test
...
Finished in 0.0227 seconds
102 examples, 0 failures
```

The suite covers:

- **Lenient pre-processor**: 32 tests, including 4 QuickCheck properties (idempotence, round-trip through aeson, explanation cardinality).
- **Schema parsers**: 11 unit tests per route (parse well-formed, fill defaults, coerce types, clamp enums, tolerate single-string-for-array, round-trip).
- **Corpus validation**: every file in `corpus/{lead-observation,prospect-audit,user-services}/` is fed through the full pipeline and must validate successfully. The corpus contains 8 + 6 + 7 = 21 fixtures covering:
  - clean JSON (control)
  - markdown-fenced JSON with leading/trailing prose
  - trailing commas in objects and arrays
  - bare (unquoted) object keys
  - smart quotes (U+2018, U+2019, U+201C, U+201D, U+00AB, U+00BB)
  - single-quoted strings
  - extra unexpected fields + trailing prose
  - placeholder text in fields (defaults layer)
- **Defaults layer**: 13 unit tests for `isPlaceholderText` + `fillDefaults*`.
- **Code generators**: 9 tests verifying the TypeScript, JSON Schema, and fixtures all have the expected structure.

## Integration plan (not implemented)

The deliverable is intentionally just the library + tests. To wire it into
the Next.js app, the recommended approach is:

1. **Build a Haskell sidecar** (using `scotty` or `warp`) that exposes
   `POST /validate` with body `{"route": "...", "raw": "..."}`. It calls
   `validateRouteText` and returns the JSON-encoded result.
2. **Modify each Next.js API route** to call the sidecar after the ZAI VLM
   call, replacing the current `JSON.parse` + fallback block.
3. **Surface warnings to the UI** so the user knows when the LLM produced
   malformed output that was silently repaired.

A rough sketch of the modified route handler:

```typescript
// In app/api/analyze-lead-images/route.ts
const content_text = response.choices?.[0]?.message?.content ?? "";
const sidecarResponse = await fetch("http://localhost:8087/validate", {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({ route: "analyze-lead-images", raw: content_text }),
});
const result = await sidecarResponse.json();
if (!result.ok) {
  return NextResponse.json({ error: result.error }, { status: 502 });
}
return NextResponse.json({
  success: true,
  images_analyzed: body.images.length,
  extracted: result.value,
  warnings: result.explanations,  // ← new: surface lenience warnings
});
```

## Design decisions

### Why Haskell?

- **Algebraic data types** make the schemas self-documenting. The
  `CallToAction` enum is a 5-constructor ADT; in TypeScript it's a string
  union that can drift from the actual JSON the LLM produces.
- **`aeson` parsers** are first-class values — we can compose them, test
  them in isolation, and accumulate errors instead of failing on the first
  one.
- **Pure functions** for the lenient pre-processor mean QuickCheck can
  property-test the round-trip ("if `extractJson` returns `Right`, the
  output is valid JSON") without any mocking.
- **Code generation from ADTs** means the TypeScript types, JSON Schema,
  and sample fixtures are all derived from the same source. No drift.

### Why not GHCJS / WASM?

Compiling Haskell to WASM and importing it into Node.js sounds appealing
(no separate process), but the toolchain (Asterius / `ghc wasm` backend)
is still experimental and has limited library support — `aeson` works, but
`aeson-pretty` and `hspec` don't. A sidecar HTTP service is simpler to
build, debug, and replace.

### Why no service layer in this deliverable?

The user explicitly asked for "Proof + tests" — a library + test suite
proving the logic is correct, with no service layer or Next.js integration.
That keeps the deliverable focused on the hardening logic, which is where
Haskell adds the most value.
